# Workflow Rules

## Branch Strategy

- Use a `main only` workflow by default unless there is a clear reason to introduce more branches.

## Commit Message Rules

Use commit messages in this format:

`<type>: <description>`

Use each type as follows:

- `feat`: when adding a user-visible feature or new capability
- `fix`: when fixing a bug or unintended behavior
- `refactor`: when improving internal structure without changing intended behavior
- `docs`: when updating documentation only
- `chore`: for maintenance work, setup changes, dependency updates, or non-feature operational changes
- `test`: when adding or changing tests
- `style`: for formatting or style-only changes with no behavior impact
